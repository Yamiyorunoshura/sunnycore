# 需求文件

## 簡介

本文件概述了Discord機器人系統全面重構和擴展的需求。該專案涉及將前端（面板）邏輯與後端（核心）邏輯分離，並實作三個新的主要模組：成就系統、經濟系統和政府系統。系統將提供面向使用者的面板和具有角色權限控制的管理介面。

## 需求

### 需求 1：架構重構

**使用者故事：** 作為開發者，我希望將面板邏輯與核心業務邏輯分離，以便系統更易於維護和擴展。

#### 驗收標準

1. 當系統重構時，所有面板相關邏輯應與核心業務邏輯分離
2. 當新增功能時，應遵循前後端分離模式
3. 當核心邏輯修改時，面板邏輯應保持不受影響
4. 當面板邏輯修改時，核心業務邏輯應保持不受影響

### 需求 2：成就系統 - 使用者面板

**使用者故事：** 作為Discord伺服器成員，我希望查看我的成就進度和獎勵，以便追蹤我的成就並查看我獲得的獎勵。

#### 驗收標準

1. 當使用者存取成就面板時，系統應顯示其當前成就進度
2. 當使用者查看成就詳情時，系統應顯示完成百分比和要求
3. 當使用者完成成就時，系統應顯示可用獎勵
4. 當使用者查看其成就時，系統應顯示已完成和進行中的成就

### 需求 3：成就系統 - 管理者面板

**使用者故事：** 作為伺服器管理員，我希望管理成就及其獎勵，以便為我的伺服器自訂成就系統。

#### 驗收標準

1. 當管理員存取成就管理面板時，系統應允許建立新成就
2. 當管理員修改成就時，系統應更新成就配置
3. 當管理員刪除成就時，系統應從系統中移除並適當處理現有進度
4. 當管理員設定成就可見性時，系統應允許隱藏/顯示使用者的成就
5. 當管理員配置獎勵時，系統應初步支援基於身分組的獎勵
6. 當管理員儲存成就變更時，系統應在應用前驗證配置

### 需求 4：經濟系統 - 使用者面板

**使用者故事：** 作為Discord伺服器成員，我希望查看我的當前餘額，以便了解我有多少可用貨幣。

#### 驗收標準

1. 當使用者存取經濟面板時，系統應顯示其當前餘額
2. 當使用者餘額變更時，系統應即時更新顯示
3. 當使用者查看其餘額時，系統應使用配置的貨幣符號和名稱顯示貨幣

### 需求 5：經濟系統 - 管理者面板

**使用者故事：** 作為伺服器管理員，我希望管理使用者餘額和貨幣設定，以便控制伺服器的經濟。

#### 驗收標準

1. 當管理員存取使用者餘額管理時，系統應允許查看和修改任何使用者的餘額
2. 當管理員變更使用者餘額時，系統應立即更新使用者帳戶
3. 當管理員配置貨幣設定時，系統應允許設定貨幣名稱和符號
4. 當貨幣設定變更時，系統應更新所有顯示以使用新配置

### 需求 6：政府系統 - 常任理事會面板

**使用者故事：** 作為常任理事會成員，我希望管理政府部門，以便維護伺服器的政府結構。

#### 驗收標準

1. 當使用者存取常任理事會面板時，系統應驗證其擁有「常任理事」身分組或伺服器管理員權限
2. 當系統檢查「常任理事」身分組且不存在時，系統應自動建立該身分組
3. 當理事會成員新增部門時，系統應在統一註冊表中建立條目
4. 當理事會成員移除部門時，系統應從註冊表中移除並處理相關帳戶
5. 當理事會成員修改部門時，系統應相應更新註冊表

### 需求 7：政府部門註冊表

**使用者故事：** 作為系統管理員，我希望有統一的部門註冊表，以便正確追蹤和管理所有政府部門。

#### 驗收標準

1. 當部門註冊時，系統應以JSON格式儲存部門名稱
2. 當部門註冊時，系統應建立或指派部門負責人身分組
3. 當部門負責人身分組不存在時，系統應在適當的部門級別身分組下建立
4. 當部門註冊時，系統應指派或建立適當的部門級別
5. 當部門級別不存在時，系統應根據使用者輸入建立身分組
6. 當部門註冊時，系統應指派部門負責人使用者
7. 當存取註冊表時，系統應以結構化格式提供所有部門資訊

### 需求 8：政府財政系統

**使用者故事：** 作為常任理事會成員，我希望理事會擁有獨立的貨幣帳戶，以便政府財政與個人使用者分離。

#### 驗收標準

1. 當建立常任理事會時，系統應為理事會建立獨立的貨幣帳戶
2. 當使用者擁有「常任理事」身分組時，系統應允許其存取和修改理事會帳戶
3. 當非理事會成員嘗試存取理事會帳戶時，系統應拒絕存取
4. 當理事會帳戶交易發生時，系統應記錄所有變更及時間戳和使用者資訊

### 需求 9：部門財政系統

**使用者故事：** 作為部門負責人，我希望我的部門擁有獨立的貨幣帳戶，以便部門財政分別管理。

#### 驗收標準

1. 當建立政府部門時，系統應為該部門建立獨立的貨幣帳戶
2. 當使用者被指派為部門負責人時，系統應授予其存取部門帳戶的權限
3. 當部門負責人存取其部門帳戶時，系統應允許查看和修改餘額
4. 當非部門負責人嘗試存取部門帳戶時，系統應拒絕存取
5. 當部門帳戶交易發生時，系統應記錄所有變更及部門和使用者資訊

### 需求 10：身分組管理整合

**使用者故事：** 作為系統使用者，我希望身分組能自動管理，以便根據職位正確分配權限。

#### 驗收標準

1. 當部門負責人身分組不存在時，系統應建立具有適當權限的身分組
2. 當部門級別身分組不存在時，系統應根據指定級別建立
3. 當建立身分組時，系統應建立適當的身分組階層
4. 當使用者被指派到職位時，系統應自動指派相應的身分組
5. 當使用者從職位移除時，系統應適當移除相應的身分組

### 需求 11：資料持久化和安全性

**使用者故事：** 作為系統管理員，我希望所有資料都能安全儲存和持久化，以便資訊不會遺失且存取受到適當控制。

#### 驗收標準

1. 當任何系統資料修改時，系統應將變更持久化到適當的儲存
2. 當系統啟動時，應載入所有現有配置和資料
3. 當執行敏感操作時，系統應驗證使用者權限
4. 當檢測到資料損壞時，系統應優雅地處理錯誤並維護資料完整性
5. 當需要備份操作時，系統應支援資料匯出和匯入功能
